Glyph-FiCoCo 优化记录（2026-06-22）

目标
- 重点修复多文档任务中的页面切分、证据保留率、跨页顺序一致性，以及长上下文证据聚合退化问题。
- 避免 FiCoCo 在 2wikimqa、hotpotqa、musique 等任务上因为过度压缩页面而丢失多跳证据。

发现的主要问题
1. 主评测链路中的页面筛选过于激进。
- 位置：glyph/inference.py
- 原逻辑按页独立打分后直接 top-k 截断。
- 风险：高分孤页被保留，但相邻证据页、首尾锚点页和跨页桥接页容易被丢掉，破坏多文档证据链。

2. 实验管线只处理渲染后的第一页。
- 位置：scripts/glyph_llava_ficoco_pipeline.py
- 原逻辑仅对 rendered_images[0] 提取视觉特征。
- 风险：多页输入时后续页面完全不参与 FiCoCo 压缩与筛选，实验分析会低估真实证据保留率。

3. 文本块切分按固定行数硬切。
- 位置：scripts/glyph_llava_ficoco_pipeline.py
- 原逻辑不识别 Document/Passage/Title/Source 等边界。
- 风险：文档边界和跨段证据被切断，长上下文证据聚合能力下降。

4. 图像化后可读性存在次生风险。
- 位置：scripts/glyph_llava_ficoco_pipeline.py
- 原逻辑对非方图使用黑底补边再缩放到 336x336。
- 风险：文本页被压缩后，小字更容易因黑边对比与整体缩放受到干扰。

已完成的优化
1. 新增 FiCoCo 顺序保留选择模块。
- 新文件：glyph_ficoco/selection.py
- 新增能力：
  - 平滑局部分数，降低单页噪声影响。
  - 邻页扩展，优先保留高分页附近的相邻证据页。
  - 区间桥接，尽量避免保留结果碎片化。
  - 锚点保护，保证受保护页面不会被低分替换掉。

2. 调整主推理链路的页面选择策略。
- 文件：glyph/inference.py
- 优化点：
  - 自动识别多文档问答提示词，如 given passages、given documents。
  - 对多文档任务启用更保守的保留策略。
  - 保护首尾页；页数较多时额外保护次首、次尾页。
  - 页选择改为“分数平滑 + 邻页扩展 + 锚点保护”，而不是纯 top-k。
  - 输出 selected_ranges 与 coverage_ratio，便于后续分析页面覆盖率。

3. 修正实验管线中的多页视觉聚合。
- 文件：scripts/glyph_llava_ficoco_pipeline.py
- 优化点：
  - 改为读取全部 rendered_images，而非只读第一页。
  - 先对多页视觉特征做页级保守聚合，再执行视觉 token 压缩。
  - 记录 selected_page_indices，便于复查被保留页面。

4. 重写文本块切分逻辑，优先保持文档边界完整。
- 文件：scripts/glyph_llava_ficoco_pipeline.py
- 优化点：
  - 先按 Document/Passage/Title/Source 等边界切成段。
  - 仅对超长段落做带重叠窗口的切分。
  - 避免原先那种按固定行数硬切造成的证据链断裂和碎片化小块。

5. 降低图像化后可读性恶化风险。
- 文件：scripts/glyph_llava_ficoco_pipeline.py
- 优化点：
  - 非方图补边从黑底改为白底，更贴近原始页面背景。
  - 这样在缩放到视觉编码尺寸时，对文本区域的干扰更小。

验证情况
- 已完成 compileall 语法检查。
- 已补充 tests/test_ficoco_selection.py 轻量回归用例。
- 已通过直接断言验证：
  - 邻页扩展策略能够保留高分页附近的辅助证据页。
  - 锚点保护能稳定保留首尾关键页。
  - 文档边界切分不再生成异常碎片块。

预期收益
- Multi-doc QA 的页面证据覆盖率提升。
- 跨页证据顺序更稳定，减少多跳链路被压断的概率。
- FiCoCo 在长上下文任务上的收益从“单纯省 token”向“尽量保留有效证据”调整。

仍需通过完整评测确认的事项
- LongBench 全量分数是否真实回升，尤其是 2wikimqa、hotpotqa、musique。
- token 压缩率可能略有回升，因为现在策略更保守，优先保证证据覆盖。


补充优化（query-guided 方案 B，2026-06-22 晚）
- 文件：glyph/retrieval.py
- 调整内容：
  - 检索改为 Unicode-aware 归一化，避免 Rákóczi、Zrínyi、cinéma 这类实体在 ASCII 分词下被拆坏。
  - 文档打分加入标题命中、短语命中、前段命中，而不是只看正文词频。
  - 低 token 截断从“纯命中句窗口”改成“标题 + 首句 + 高命中句”保留，减少桥接证据被截成尾部碎片。
  - 选择逻辑加入覆盖增益，优先选能补齐未覆盖查询实体的段落，而不是单纯 top-k。

- 文件：evaluation/longbench/scripts/post_api_longbench.py
- 调整内容：
  - query-guided 渲染不再用完整 VLM prompt 做检索，改为优先使用原始问题字段 input/question。
  - 保存 retrieval.selected_text 和 retrieval_query，便于复查证据保留与图像化可读性。
  - 轻度放宽紧缩预算：max_sentences_per_unit 从 4 调到 5，max_chars 从 2200 调到 2600。

- 文件：scripts/word2png_function.py
- 调整内容：
  - config_dict 分支补齐 font-path 解析。
  - 当数据中的相对字体路径无法直接解析时，优先回退到仓库内 Verdana.ttf，再回退到系统 DejaVuSans.ttf。
  - 这个修复是必须项，否则 Gen_env 下 query-guided 动态渲染会在 ReportLab 阶段因字体未注册而中断。

最新结论
- 在约 290 prompt tokens 的压缩预算下，query-guided 方案已经明显优于旧版极端压缩 FiCoCo 基线，且生成的单页 PNG 高度大多在 276-346 像素区间，仍保持可读。
- 但当前 Glyph 模型栈在不同多文档任务上的最优压缩策略并不统一：
  - 2wikimqa、musique 可以在低 token 下得到明显改善。
  - hotpotqa 在同样预算下反而退化。
- 这说明当前瓶颈已经不主要是“页面切分/渲染可读性”，而是“检索质量不足以稳定支持所有多跳证据链”。
- 如果目标是继续以压缩优先为主，下一步不应再盲目收紧页面，而应升级检索器（BM25 / dense / entity linking）或直接更换更强 VLM。
