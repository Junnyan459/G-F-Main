Glyph-FiCoCo 多文档采样复测记录（Gen_env，2026-06-22）

复测目的
- 验证本次页面保留与边界切分优化，是否真的改善了最关键的多文档退化任务。
- 优先检查旧版退化最明显的三个数据集：hotpotqa、2wikimqa、musique。

复测范围
- 环境：/root/miniconda3/envs/Gen_env
- 输入：/root/autodl-tmp/Data/longbench/rendered_images 中的 hotpotqa、2wikimqa、musique
- 新结果目录：/root/autodl-tmp/Data/longbench/pred/gen_env_ficoco_opt_multidoc10_20260622
- 旧结果目录：/root/autodl-tmp/Data/longbench/pred/gen_env_ficoco_full
- 评测方式：每个数据集只取前 10 条样本进行对比，使用 LongBench 原评测脚本中的 scorer 计算分数。

结果对比（前 10 条样本）
1. hotpotqa

- 新版 top10 分数：28.0
- 分数变化：+8.0
- 旧版平均 prompt tokens：54.3
- 新版平均 prompt tokens：2837.6
- token 变化：+2783.3

2. 2wikimqa
- 旧版 top10 分数：22.5
- 新版 top10 分数：57.74
- 分数变化：+35.24
- 旧版平均 prompt tokens：49.7
- 新版平均 prompt tokens：2093.6
- token 变化：+2043.9

3. musique
- 旧版 top10 分数：10.0
- 新版 top10 分数：20.0
- 分数变化：+10.0
- 旧版平均 prompt tokens：50.2
- 新版平均 prompt tokens：3424.9
- token 变化：+3374.7

结论
1. 从最关键的多文档采样结果看，这次优化方向是有效的。
- hotpotqa、2wikimqa、musique 的前 10 条样本分数全部提升。
- 其中 2wikimqa 提升最明显，说明之前最严重的证据丢失问题确实被缓解了。

2. 这次提升的代价非常高。
- prompt token 使用量大幅反弹，说明当前策略已经从“压缩优先”转向“覆盖优先”。
- 换句话说，效果变好主要来自保留了更多页面和更多跨页证据，而不是在相同 token 预算下更高效地压缩。

3. 当前版本不能直接视为最终最优方案。
- 如果目标是修复多文档退化，当前修改是有效的。
- 如果目标是同时保持 FiCoCo 的低 token 优势，当前版本还不够平衡，需要进一步调参。

建议
1. 下一步应在完整 LongBench 上继续复测，但优先关注 Multi-doc QA 类别，而不是先跑全量所有类别。
2. 需要继续压回 token 成本，建议优先调以下参数：
- ficoco_keep_ratio
- ficoco_min_pages
- ficoco_max_pages
- 多文档任务下的邻页扩展半径
- 首尾锚点保护数量

3. 一个可行方向是做分任务策略：
- 对 hotpotqa、2wikimqa、musique 这类多跳任务使用更保守的页面保留。
- 对其他任务继续使用更激进的压缩配置。

限制说明
- 本次只是关键多文档数据集的前 10 条样本采样复测，不是完整 LongBench 全量结果。
- 因此当前结论可以说明“优化方向有效”，但还不能直接替代完整 benchmark 结论。


补充复测（query-guided compact fix，Gen_env，2026-06-22 晚）

复测背景
- 旧的紧缩版 query-guided 虽然把 prompt token 压到约 244-251，但 2wikimqa 从 22.5 跌到 2.5，已经失效。
- 排查后确认主要问题有三类：
  - 检索用的是完整提示词，不是原始问题。
  - ASCII 分词会破坏带重音实体。
  - 动态 query-guided 渲染在 Gen_env 下会因相对字体路径未解析而崩溃。

本轮修复后的结果（前 10 条样本）
1. hotpotqa
- 旧版 FiCoCo 基线：20.0，54.3 tokens
- 旧紧缩 query-guided：34.0，244.2 tokens
- 新紧缩 query-guided fix：17.5，288.3 tokens
- 结论：在更稳的多跳保留策略下，hotpotqa 没有保持住之前的高点，说明当前轻量检索策略对该任务不稳定。

2. 2wikimqa
- 旧版 FiCoCo 基线：22.5，49.7 tokens
- 旧紧缩 query-guided：2.5，243.4 tokens
- 新紧缩 query-guided fix：29.4，289.6 tokens
- 结论：修复后已经明显高于旧版 FiCoCo 基线，也显著修复了旧紧缩 query-guided 的灾难性退化。

3. musique
- 旧版 FiCoCo 基线：10.0，50.2 tokens
- 旧紧缩 query-guided：16.67，250.6 tokens
- 新紧缩 query-guided fix：25.0，294.7 tokens
- 结论：在仍然较低的 token 预算下继续提升，说明 query-guided + 可读性约束渲染对该任务有效。

整体判断
1. 这轮修复证明：
- query-guided 压缩不是方向错误。
- 真正的问题是“检索器太弱”以及“动态渲染链路不稳定”。

2. 当前最平衡的观察结论是：
- 与旧版极端压缩 FiCoCo 基线相比，新紧缩 query-guided fix 在 2wikimqa 和 musique 上确实实现了“低 token 下的真实提升”。
- 但它没有在所有多文档任务上同时提升，尤其 hotpotqa 仍然不稳定。

3. 因此当前不能宣称已经实现“统一有效的压缩最优解”。
- 如果必须维持大约 300 prompt tokens 级别的强压缩，当前 Glyph 栈还不够稳。
- 下一步建议二选一：
  - 升级检索：BM25 / dense retrieval / entity-aware reranking。
  - 更换更强 VLM：让图像读段能力和跨页证据聚合能力一起提升。

4. 是否真正提升
- 相对旧版 FiCoCo 基线：部分任务真正提升，尤其 2wikimqa、musique。
- 相对旧版紧缩 query-guided：修复后整体更稳，不再出现 2wikimqa 近乎失效的问题。
- 相对“所有任务都提升”的目标：尚未达到。
